## Pre-insertion planning checklist

Complete all items before treating the pack as ready for repo-native insertion/adoption:

1. Re-check the live repo truth surfaces listed in the working-use plan.
2. Confirm the pack still reflects the intended Lane Class A vs Lane Class B boundary.
3. Confirm the representative fixture lock still matches the live fixture manifest.
4. Confirm the current pinned Candidate A artifact is still the correct historical workbench/before-state reference artifact and is not being mistaken for the controlling definition of admitted integrated behavior.
5. Classify each doc as active control, operational companion, or archive/context.
6. Record the intended status relationship for each doc to be inserted: `supplements`, `extends`, `parallel working artifact`, or `non-authoritative support`.
7. Prepare the final filename / ordinal mapping for every doc intended for adoption.
8. Check for ordinal collisions, filename collisions, or front-door/navigation collisions against the current live repo control spine rooted at `next_milestone_plans/multi_variant_visual_lane_control/README_INDEX.md`.
9. Confirm all pack-local markdown references still resolve before any filename or ordinal changes are applied.
10. Confirm the intended adoption relationship against the retained-default authority chain remains explicit:
   - `next_milestone_plans/multi_variant_visual_lane_control/03AC_EXACT_POST_ADMISSION_DEFAULTING_SCOPE_AND_DECISION_BOUNDARY.md`
   - `next_milestone_plans/multi_variant_visual_lane_control/05O_POST_ADMISSION_DEFAULTING_PLANNING_FREEZE_PACKET.md`

## Insertion / adoption checklist

Complete these items during the formal insertion/adoption step:

1. Apply the approved final filename / ordinal mapping only to the docs being adopted.
2. Update `next_milestone_plans/multi_variant_visual_lane_control/README_INDEX.md` and any adopted repo-native navigation/front-door surfaces for the adopted set.
3. Mark operational docs as non-normative and ensure the normative-doc set is explicit.
4. Preserve archive/context docs as clearly non-authoritative.
5. Verify that no rule exists only in an operational companion doc at insertion time.
6. Verify the adopted docs supplement or extend the retained-default authority chain rather than silently displacing it by implication.

---

## Post-insertion verification checklist

Complete all items after insertion/adoption is applied:

1. Verify all adopted links and references resolve.
2. Verify no stale filenames, ordinal references, or standalone-only wording remain where adopted names should now be used.
3. Verify front-door/navigation inclusion for the adopted docs, including the live repo front door in `next_milestone_plans/multi_variant_visual_lane_control/README_INDEX.md`.
4. Verify authority wording is explicit and does not overclaim repo-native authority before adoption or understate it after adoption.
5. Verify no doc now implies broader runtime, OCR/media, or schema widening.
6. Verify no duplicate active-control-doc sprawl or parallel-rule source has been introduced.
7. Verify the same-selector-only-if-equivalent rule, representative-fixture rule, and Lane Class A boundaries remain unchanged.
8. Record any unresolved contradiction explicitly rather than smoothing it over.

---

## Result

The pack is ready for disciplined repo-native use only when the pre-insertion, insertion, and post-insertion sections are all clean.
